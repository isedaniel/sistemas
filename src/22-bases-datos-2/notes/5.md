# Alta disponibilidad

Se trata de mantener la continuidad del servicio. Es decir, minimizar el tiempo
de caída de los servicios.

Cada alternativa de alta disponibilidad tiene sus propias particularidades.

Cada motor tiene sus propias funciones de alta disponibilidad.

# Opciones de alta disponibilidad

- Failover Cluster Instances (FCI)

- Alway On Availability Groups

- Database Mirroring

- Log Shipping

- Replication

Se diferencian en:

- Failover Automático: Cuando la instancia principal se cae, esta opción
  automáticamente pasa el servicio a otra instancia.

- Lectura secundaria: cuando una opción de alta disponibilidad cuenta con dos
  instancias, esta característica determina si la instancia secundaria puede
  ser utilizada o no para lectura.

- Edición mínima de SQL Server: Standard, Enterprise o Express+.

- Tipo: puede ser propio de la instancia, de la base de datos dentro de la
  instancia, o de un objeto específico.

# Failover Cluster Instance (FCI)

Dos servidores de SQL Server pueden tener un Storage (servidor que maneja
discos) en común. Solamente uno de los servidores tiene acceso a los discos.

El otro es pasivo. Si el activo se cae, el servidor 2 detecta esa caída y toma
el control de la instancia. Y tiene acceso al Storage común.

Pueden ser físicos o virtuales. Con discos virtuales.

Una instancia se puede ejecutar en varios nodos (servidores) en un clúster
Windows.

Cada servidor tiene los binarios. Pero el almacenamiento es uno solo. Como los
discos son los mismos, los datos no cambian.

Tenemos alta disponibilidad a nivel Hardware.

Si se rompe el Storage, perdemos todo. Si tenemos un ARRAY podemos tener la
información en más de un Disco y no se pierde la información. Un disco
defectuoso se cambia y listo. Esos discos puede ser _HOT SWAP_.

El _cluster_ tiene una IP propia flotante. Que puede ir al nodo 1 o nodo 2. No
en ambos a la vez.

## Ventajas

- Alta disponibilidad automática

- _Transparente_ para las aplicaciones. No importa el nodo activo para el
  usuario.

## Consideraciones

- No soluciona el balanceo de carga: el otro nodo es pasivo. No puede
  conectarse a la base de datos en el Storage.
- Requiere configuración compleja y recursos compartidos (Storage)
- Requiere licencia, aunque pagas una licencia por el Cluster.

# Always ON

Característica de SQL Server. Reemplaza al _MIRROR_. Es muy ágil. Flexible,
distintos tipos de configuraciones.

Puede tener varios nodos. Cada nodo tiene su propio disco, a diferencia del
Failover Cluster.

Cada nodo tiene sus binarios y sus discos.  Por ello utiliza más espacio de
disco porque tienes _N-Storage_.

Un nodo es el pricipal. El resto replican del principal.

Se pueden utilizar estando todos dentro del mismo _datacenter_. O teniendo uno
de estos nodos en un datacenter externo.

Algunas versiones de SQL Server lo pueden tener en la nube.

Igual que _Failover Cluster_: solo un nodo está activo. La IP es flotante. Solo
el nodo activo la utiliza.

Tiene _Failover automático_. En caso de fallo toma la IP. Pero no los discos,
porque cada uno tiene su storage.

Soporta hasta 8 réplicas.

Soporta lectura secundaria pero depende de una licencia adicional.

## Ventajas

- Alta disponibilidad y sin interrupción perceptible. Las instancias siempre
  están levantadas. Porque están sincronizando. A diferencia de _FAILOVER
  Cluster_, que tiene unos minutos hasta que levanta la otra instancia.
- Permite diferentes sitios geográficos
- Solo pagamos una licencia.

## Consideraciones

- Podemos pagar más de una licencia si tenemos lectura secundaria: un nodo
  pasivo se usa para lectura. Es una distribución de carga de solo lectura.
- Requiere de _Windows Server Failver Clustering_ para la IP flotante.
- Requiere SQL Server Enterprise.
- Tiene mayor complejidad.

# SQL Mirroring

Es la versión anterior de _ALWAYS ON_. También requiere de por lo menos dos
instancias. Cada una replica entre un servidor y otro. Es decir, cada uno
necesita su propio _STORAGE_. Igual que _ALWAYS ON_.

Mantiene una copia idéntica de las bases de datos en ambos nodos.

Failover cluster mantiene disponibilidad a nivel de toda la instancia. La base
está en el mismo disco.

Tanto _Mirroring_ como _ALWAYS ON_ es para cada una de las bases de datos. No
para toda la instancia. Esto es importante porque un Cluster requiere que todos
los nodos soporten toda la instancia. En cambio aquí podemos tener diferentes
criterios por base de datos, porque una es más crítica que la otra. Por lo
tanto, podemos tener un servidor potente y otros de menor rendimiento que
repliquen ciertas bases de datos.

Tengo un servidor principal. Después tengo centros de contigencia, que
sincronizan la copia de seguridad del principal.

Puede operar en:

Alta seguridad. La información al servidor secundario va cuando hemos hecho
`COMMIT` (no permite lectura sucia). Los `COMMIT` son sincrónicos: se hace
`COMMIT` en el nodo principal y en las réplicas.

Alto rendimiento. (Asincrónico). Primero se hace `COMMIT` al principal, luego
se envía al resto.

Automático. Si tiene o no _FAILOVER AUTOMÁTICO_.

Requiere que las bases de datos estén en modo full (se guarda tanto la base
como los logs).

## Ventajas

- Alta disponibilidad rápida
- Tiene failover automático
- Más fácil de configurar que ALWAYS ON y FAILOVER CLUSTER
- No requiere almacenamiento compartido

## Consideraciones

- Desaprobado desde SQL Server 2016, todavía funciona pero lo van a dar de
  baja.
- No permite lectura secundaria en los mirrors.
- Solo funciona con una base a la vez.
- Requiere sincronización constante, exigente en rendimiento.

# Log Shipping

Es el más viejo y engorroso.

Tenemos una base primaria, una base secundaria. Cuando uno confirma una
transacción, se guarda en la base y en el log.

Tenemos un backup de log cada 5 minutos. Cada vez que se genera un backup de
log, se copia del servidor primario al secundario.

En el secundario se hace un _RESTORE_ del Log a la base de datos del servidor
secundario.

Una vez que tengo las bases iguales, puedo activar el Log Shipping, de la
principal activa a la secundaria pasiva.

## Ventajas

- No necesitas bajar la base de datos para nada. Solo alcanza con configurar el
  _backup full_.
- Con dos servidores en la misma red se arma fácil.

## Consideraciones

- No se sincronizan constantemente. Sino en el intervalo que configuremos.
La diferencia entre la primaria y secundaria es como mínimo de 5 minutos.
Es muy probable que perdamos información si se cae la instancia principal.
- Es engorroso de mantener porque hay que copiar LOGs a mano.
- No se puede poner el pasivo como activo y que sincronice automáticamente. Hay
  que volver a armar el sincronismo.
- No está demasiado automatizado.
