# Gestión de bloqueos

Característica de los motores de bases de datos. Colaboran con la integridad de
la base de datos.

Tiene como objetivo proteger las transacciones cuando _ingresan_. Aseguran
mantener el aislamiento cuando el usuario lo están ingresando a los datos.
Asegura que estos cambios no son vistos por otros.

Solo se ven los guardados y confirmados. Cuando unor realiza el `COMMIT`.

Uno puede abrir una transacción. Hacer cambios. Pero esos cambios no se ven
hasta el `COMMIT`.

Puede terminar con `COMMIT`: se confirma y se guarda. EL resto de usuarios
pueden ver. Sino puede terminar con `ROLLBACK`: no se guardan los cambios.

Lo que todavía no está confirmado, el resto de usuario no lo pueden ver. Las
sesiones generan bloqueos sobre los datos con que están trabajando.

Esos bloqueos pueden ser modo **compartido** o modo **exclusivo**.

# Introducción

Solicito bloqueo y hay conflicto. El motor pausa la transacción hasta que
finaliza el primer bloqueo.

Ejemplo: hago un update sobre cliente. La base bloquea la información para que
otros no puedan modificar el dato.

Otros pueden ver el registro, pero no ver mi cambio. Tampoco puede borrarlo o
modificarlo. Para eso sirven los bloqueos.

Depende de la configuración del nivel de bloqueo. Para generar un bloqueo tiene
que hacer algo: `UPDATE`, `DELETE`. Pero no hacer `COMMIT`.

El procesador de consulta determina el tipo de bloqueo que se necesita para
proteger cada recurso.

El bloqueo puede ser de bloque o de la tabla completa.

# Clasificación de bloqueos

- Pesimista. Los demás usuarios no pueden realizar acciones que creen
  conflictos hasta que el propietario lo libere.

- Optimista. Cuando un usuario actualiza, el sistema comprueba si el otro ha
  cambiado los datos. Si el otro también cambió, se produce un error. El
  usuario que recibe el error revierte la transacción y arranca de nuevo.

En `SQL Server` `SELECT` también es bloqueante. Por eso las consultas tienen
que estar optimizadas. Para que el bloqueo dure la menor cantidad de tiempo
posible.

# Niveles de aislamiento

- Read committed. Nivel de aislamiento predeterminado. Las intrucciones
  `SELECT` puede hacer bloqueos compartidos. Establece mecanismos para lectura
  que datos que obtengan valores confirmados.

# Nivel 2. Read uncommitted.

No establece bloqueo compartido. Permite leer a pesar de la existencia de
bloqueos. Sin embargo, podrían leer datos no confirmados, que puede sufrir un
`ROLLBACK`. Esto se conoce como **lectura sucia**.

```SQL
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

SELECT ...
```

En otra sesión:

```sql
BEGIN TRAN;

UPDATE ...
```

Aunque no haga `COMMIT`, el usuario 1 puede ver los cambios en el `UPDATE` de
la sesión 2. Si después hace un `ROLLBACK`, la sesión 2 volverá a ver el dato
anterior. Por eso se denomina **lectura sucia**.

Se usa de manera muy puntual. Cuando necesito evitar que se bloquee la tabla. O
bloquear otro procesos. No se puede usar para facturación. Produciría errores
groseros.

# Nivel de aislamiento 3. Repeatable Read.

Hace bloqueo compartido hasta que finalice la transacción.

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

SELECT ...
```

Tengo una segunda conexión:

```sql
BEGIN TRAN;

UPDATE ...
```

Con una tercera conexión.

- Nivel de aislamiento 5

Lectura repetible. Mientras no se confirme la transacción en conexión 1,
conexión 2 puede leer. Pero no verá los datos actualizados.

Esto es lo normal.

# Lectura fantasma

Cuando el número de filas es distinto en dos lecturas consecutivas.

# Nivel aislamiento 6

Es el aislamiento más estricto. Todas las lecturas son repetibles. No se
permiten nuevas filas.

```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE;

SELECT ...
```

# Nivel de aislamiento 7

**SNAPSHOT**. Evita problemas de lecturas repetidas y sucias. Propio de
Microsoft. Guarda versiones en `TEMPDB`.

Se activa con `SET TRANSACTION ISOLATION LEVEL SNAPSHOT;`.

Si esto activa el usuario 2. Por más que otro haga `COMMIT`, va a seguir viendo
el `SNAPSHOT`. Es decir, no veo _ni_ los `COMMIT`. Solo el `SNAPSHOT`.

# Modos de bloqueos

En cada transacción podemos ver el modo de bloqueo de datos. Tenemos rangos de
bloqueo.

# Bloqueo compartido

Utilizados para sentencias que no modifican datos: `SELECT`. Se representa con
`request_mode="S"`. Permite que varias transacciones simultáneas lean un
registro. En simultaneidad pesimista.

Otras transacciones no pueden modificar los datos mientras el bloque compartido
exista.

Importante a tener en cuenta para las aplicaciones.

# Bloque exclusivo

Evita que transacciones simultáneas accedan al recurso.

Generado por `UPDATE`, `DELETE`, etc.

Representado por `request_mode="X"`.

# Bloqueo de actualización

Representado por `request_mode="U"`. Evitan el interbloqueo.

En las transacciones de lectura adquiere `S`. Se pasa de un `S` a un `X`. En
caso que sean 2, se genera un bloqueo compartido. Pero si una de ellas quiere
generar un bloquero exclusivo, una tiene que esperar.

# Bloqueo de intención

Usa las transacciones para declarar su intención de declarar un bloqueo
compartido. Es para que se solicite antes que los `S` y `X`.

Mejoran el rendimiento. Porque se determinan al nivel de tabla.

Elimina la necesidad de analizar cada bloqueo de fila o tabla.

# Bloqueo de esquema

Se usa cuando se modifica una tabla agregando una columna. O se elimina una
tabla.

Evita acceso simultáneo. Bloquea todas las operaciones externas. El motor usa
este al compilar y ejecutar consultas. No impide que los transaccionales se
realicen. Un X o S pueden correr.

# Compatibilidad de bloqueo

Hay bloqueos que pueden solicitarse concurrentemente y otros no.

# SQL Server

Con `Activity Monitor` podemos ver las sesiones conectadas a la instancia de
SQL Server.

## Query 1

```sql
BEGIN TRAN;
UPDATE Estudiantes
SET edad = 50
WHERE idEstudiante = 1;
```

Si selecciono la `query`, y ejecuto _sin_ la instrucción TRAN. Se hace
inmediatamente.

Si incluyo `BEGIN TRAN`, pero sin hacer `COMMIT` o `ROLLBACK`, la transacción
todavía no se cerró.

El monitor de transacciones muestra que la sesión bloquó la tabla. Porque está
haciendo un `UPDATE`. Y todavía no corrió `COMMIT` ni `ROLLBACK`.

Puedo correr, desde la segunda sesión, la query:

```sql
SELECT edad
FROM Estudiantes
WITH (NOLOCK) -- especifico que quiero hacer una lectura sucia
WHERE idEstudiante = 1;
```

Como incluyo la sentencia `WITH (NO LOCK)`, le indico al motor que quiero hacer
una _lectura sucia_. Es decir, voy a _leer el dato_ como _estaría_ si el
usuario 1 hace `COMMIT`. El riesgo lógico que corremos con esto es que puedo
leer un dato al que luego se le haga `ROLLBACK`.

El `WITH` tiene la diferencia que no cambia toda la sesión para un determinado
comportamiento frente a los bloqueos. Sino que cambia el comportamiento frente
al bloqueo para la _query_ es la que lo llamamos.

# Copia de seguridad

**Por qué.** Para mantener la integridad de los datos.

Hasta ahora vimos que se grabe como el negocio lo necesite. Para eso son las
restricciones. Que un DNI no esté duplicado. Que dos usuarios no tengan el
mismo mail. Eso sería la integridad lógica.

Con respecto a la integridad física, viene el problema de los _backups_.
También es interesante frente a errores humanos, `DELETE` mal hecho, un
`TRUNCATE`. El _backup_ nos protege de estos problemas.

(Notar que la `FOREIGN KEY` también impide el `DROP` de la tabla).

La integridad física puede ponerse en riesgo por problemas de Hardware,
desastre naturales.

# Políticas de respaldo

Conjunto de directrices o procedimientos para hacer _backup_.

Por ejemplo: Si son completos. A qué hora se hacen. Si tengo otros centros de
cómputos. Si tengo réplicas de datos. Temas regulatorios.

En concreto, hay que considerar:

## Objetivo del respaldo

Finalidad y objetivos específicos: si es disponbiilidad, protección frente a
corrupción de datos, cumplimiento de normativas (por ejemplo, datos si o si
dentro del territorio).

## Tipo de respaldo

Especifica tipos de _backups_: Completos, incrementales, diferenciales, de
registro de transacciones.

## Frecuencia

Con qué regularidad se realizan. Diaria, semanal, mensual, en tiempo real,
dependiendo de la criticidad de los datos.

## Almacenamiento

Dónde en qué medio se almacenarán (local, remoto, en nube, cintas) y cómo se
protegeran en dicho medio.

## Política de retención

Cuánto tiempo mantengo los datos. Puede depende de muchos factores.Como
políticas de la empresa o disposiciones de entidades regulatorias. Ejemplo: el
BCRA pone un plazo mínimo para el mantenimiento de datos.

## Procedimientos de restauración

Pasos a seguir para restaurar los datos. Incluye pruebas para testear
efectividad de la restauración.

## Documentación y auditoría

Cómo se documenta, monitorea y audita la política de _backups_.

## Plan de recuperación

Ante un _desastre_, cuál sería el plan. Por ejemplo, si se rompe el
equipamiento de un centro de cómputo de la compañía. Es más importante en
grandes organizaciones.

# Estrategias de copias de seguridad

Importante tener en claro el medio de los _backups_. Es importante, por ello,
el Hardware específico. Cuánto tiempo demora depende del medio.

Quién es el responsable.

Minimizar la pérdida de datos.

Hay que hacer también _backups_ del sistema: información del sistema operativo,
propia de la instancia, como los usuarios autorizados a ingresar.

El ítem más importante es minimizar la pérdida de datos.

El segundo ítem más importante es la _disponibilidad_ del sistema. A mayor
criticidad de la base de datos para nuestro negocio, mayores serán las
inversiones en tiempo y dinero que tendremos que hacer.

# Tipos de backup

## Backup en caliente

Se hace en tiempo real, sin dar de baja el servidor. La ventaja es que no es
necesario bajar el sistema. La desventaja es que requiere mayor poder de
cómputo.

## Backup en frío

Es lo contrario: requiere bajar la instancia de `SQL Server` para hacerse.

La desventaja es que interrumpimos el servicio momentáneamente.

Lo bueno es tiene un menor impacto en el rendimiento. Además, está garantizada
la consistencia porque no hay transacciones activas en el momento del _backup_.

# Archivos de una base de datos

## Archivo primario de datos .mdf

Aquí se generan las tablas e índices. Se guardan los datos. Es _realmente_ la
base.

## Archivo secundario de datos: .ndf

Es el archivo _log_. Va registrando los cambios que se hacen.

## Extent: 8 páginas contiguas de 8KB

## Archivo de registro de transacciones: .ldf

Aquí se guardan los datos viejos y nuevos. Por ejemplo, mientras estamos
haciendo transacciones. De aquí se toman las _lecturas sucias_. Si hacemos
`COMMIT`, pasa al .mdf. Si hacemos `ROLLBACK`, no tocamos el .mdf.

# Backup frío vs caliente

Al frío no le puedo adicionar información de otros backups.

Al caliente le puedo adicionar transacciones posteriores al momento en que se
hizo ese backup en caliente.

De esta manera, al _backup full_ le puedo adicionar el _backup_ de los _logs_.
Así pierdo la menora cantidad posible de datos. Minimizamos la pérdida de
datos.

# Modelo simple

```sql
BACKUP DATABASE
  AdventureWorks
TO DISK='<PATH>'
WITH FORMAT;
GO
```

Ejemplo de _backup caliente_. El modelo simple permite limpiar el _log_. O
ponerle un tamaño máximo a partir del cual se limpia.

No hacemos backup de logs en modelo simple.

# Modelo completo

Requiere copia de seguridad de los registros. No se pierde.

```sql
USE master
ALTER DATABASE AdventureWorks SET RECOVERY FULL;
```

Seteamos modelo completo con `SET RECOVERY FULL`.

Hacemos _backup full_ y después hacemos _backup de logs_. Este guarda la base y
además los _logs_.

# Backup por medio de registros de operaciones masivas

Modo _Bulk-logged_. Similar al full, pero minimiza el uso del log de
transacciones durante operaciones masivas.

Registra qué páginas fueron modificadas, pero no registra cambios por fila.

# Backup full diferencial

Genera un full primero. Después backup de logs, con las transacciones
posteriores.

A la hora de hacer un nuevo backup full, hace un **backup diferencial**, que es
un backup de todos los logs.

Puede ser una cinta. Tiene que ser compatible con Windows.
