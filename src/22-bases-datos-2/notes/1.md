# Integridad de datos

Refiere a la corrección de los datos. Que va a ser completa y fiable.

Garantiza que los datos estén a salvo de fuerzas externas.

Una de las funciones de la BD relacional es preservar la integridad de los
datos.

No es solo que el ingreso de datos sea correcto. También es importante que se
preserve su integridad.

# Tipos de integridad

1. Integridad Física

Protección de los datos tal como están almacenados. Ejemplo: Frente a
catástrofes, hackers, apagones.

Los motores nos dan mecanismos para resguardar la integridad física. Con
_backups_, _log backups_, _DRS_.

Activar un DRS es activar un centro de cómputos paralelo, donde las bases de
datos se replican constantemente, de forma sincrónica o asincrónica. Frente a
una contingencia se activa el DRS. (Entiendo, Data Replication System).

Las políticas concretas dependen de las necesidades y recursos de la
organización.

Es decir, en la integridad física lo fundamental son los _backups_ y los
sistemas.

2. Integridad Lógica de los datos

Protección del error humano o de un actor malicioso (_hacker_), de una forma
distinta a la integridad física.

Consiste en establecer reglas para que los datos sean confiables y la
integridad de la base de datos se preserve en el tiempo.

Ejemplo: en la tabla Clientes, el campo localidad tiene una propiedad
`NOT NULL`, que impide que guardemos un cliente sin especificar su localidad.

Es decir, en la integridad lógica lo fundamental son las restricciones que
establecemos.

# Integridad lógica de datos

Cuando modificamos datos con sentencias `INSERT`, `DELETE`, `UPDATE`, la
integridad de datos puede perderse. Por ejemplo, añadiendo o modificando datos
no válidos. Además, los cambios pueden perderse por un fallo en el suministro
eléctrico.

Antes de hacer efectivo un `COMMIT`, el motor chequea que se cumplan las
restricciones. Sino, tocal `ROLLBACK`.

Para eso se establecen restricciones lógicas.

# Restricciones lógicas

* Restricciones por `PRIMARY KEY`

Solo podemos tener un `ID` único para cada registro.

* Restricciones referenciales

No podemos tener una referencia inválida, es decir, no refernciar un `ID` que
no exista.

* Restricciones de dominio

Un valor no puede salirse de un rango determinado.

* Restricciones Default

Tenemos un valor por default para evitar valores `NULL`.

* Restricciones `NOT NULL`

No admitimos `COMMIT` con valores `NULL`.

* Restricciones `UNIQUE`

Cada registro debe tener un valor único en dicho campo.

* Control de transacción (comando `TCL`)

Si se modifica un registro, qué datos queremos guardar de esa transacción.

* Disparadores (`TRIGGERS`)

Se dispara ante un mensaje para cuidar la integridad. Cuando se hace un
`COMMIT`, qué _metadatos_ se guardan (_logs_).

Esto no solo depende del motor de base de datos (RDBMS).

# Restricción `PRIMARY KEY`

Identifica una o más columnas como _clave principal_. Se permite solo una
`PRIMARY KEY` por tabla. Esto significa que el _campo_ debe ser único para cada
_registro_. Además, la `PRIMARY KEY` no puede ser `NULL`.

```sql
CREATE TABLE Facturas (
    numero INT IDENTITY(1000, 1) NOT NULL,
    fecha DATETIME,
    CONSTRAINT pk_historico PRIMARY KEY (numero)
)
```

Ponemos nombre a la `CONSTRAINT` para reconocerla frente a un eventual error.

# Restricciones referenciales

Restricciones por `FOREIGN KEY`, para garantizar integridad entre columnas,
de la misma o distintas tablas.

Debe referenciar una `PRIMARY KEY`

El usuario que la arma debe tener _permiso_ `REFERENCES`

# Restrición de dominio

Establecemos un rango para un campo.

```sql
CHECK campo > 10 AND campo < 50
```

# Restricciones `DEFAULT`

Establecemos un valor predeterminado para cuando no se proporciona un dato:

```sql
ALTER TABLE pedidos ALTER fecha SET DEFAULT '2002-01-01';
```

# Restricciones `NOT NULL`

Establezco que un campo no puede ser nulo a la hora de cargar los datos.

# Restricciones `UNIQUE`

Asegurar que el valor de un campo es único para cada registro. La restricción
`UNIQUE` presupone una política para generar `ID`.

# Restricción diparador (`TRIGGER`)

Es un tipo de _Stored Procedure_, que se ejecuta cuando se cumple una
condición.

Ejemplo: Cuando se modifica/agrega un registro. Cuando se modifica una tabla.

```sql
CREATE TRIGGER nombre ON tabla
    FOR <insert/update/delete>
    AS
        <sentencias>
```

Pueden incidir sobre la performance, por lo que deben usarse con juicio.
