# Autenticación vs Autorización

Autenticación. Validar que el cliente es quien dice ser.

Autorización. Las cosas que podemos hacer o no. Que me autentique no implica que
estoy autorizado a hacer todo lo que quiero. Se relaciona con el _rol_.

Creamos un proyecto con el template de `webapi`.

```sh
dotnet new webapi -o Seguridad --use-program-main --use-controllers --no-openapi
```

`--use-controllers`. Permite usar el patrón de diseño MVC.

`--no-openapi`. Permite usar Swagger y autodocumentar. No lo vamos a usar en
nuestro proyecto.

Corremos el proyecto.

```sh
dotnet run # => Levanta server en http://localhost:5136/WeatherForecast
```

Limpiamos el método `Main`.

```csharp
public class Program
{
    public static Main(string[] args)
    {
        var builder = WebApplication.CreateBuilder(args);
        
        builder.Services.AddControllers();
        
        var app = builder.Build();

        app.UseHttpsRedirection();
        app.UseAuthorization();

        app.MapControllers();
        app.Run();
    }
}
```

Servidor. Un proceso escuchando en un puerto. Recibe solicitudes `http` que
siguen un estándar. Y envía respuestas a esas solicitudes.

ASP.NET y Express usan el concepto de Middleware. Son métodos que procesan las
solicitudes que llegan. Por eso se denomina _Pipeline_.  Cada método es una
etapa en esa _tubería_.

El concepto de Middleware implica un orden. Se ejecuta un método después del
otro. El orden es relevante.

Agregamos servicios para autenticación y autorización.

```csharp
public class Program
{
    public static Main(string[] args)
    {
        var builder = WebApplication.CreateBuilder(args);
        
        builder.Services.AddControllers();
        // Agregamos los servicios de autenticación y autorización
        builder.Services.AddAuthentication();
        builder.Services.AddAuthorization();
        
        var app = builder.Build();

        app.UseHttpsRedirection();
        // Agregago UseAuthentication a nuestro pipeline
        app.UseAuthentication();
        app.UseAuthorization();

        app.MapControllers();
        app.Run();
    }
}
```

Vamos a usar un toker JWT. Un estándar. Para configurar vamos a
`appsettings.json`.

```js
{
    // ...
    "JwtSettings": {
        // Una semilla para la encriptación
        "SecretKey": "asdf;lkjqwpoiuwerq;lkajsdfpoiquwer;lkjzcva;sdjkfpqoiwer",
        // Somos nosotros
        "Issuer": "http://localhost:5136",
        // Nosotros de nuevo
        "Audience": "http://localhost:5136"
    }
}
```

Agregamos el paquete de `jwt`.

```sh
dotnet package add Microsoft.AspNetCore.Authentication.JwtBearer
```

Este comando descarga el paquete a `$HOME/.nuget` (si no lo teníamos) y lo
agrega  a `Seguridad.csproj`.

Configuramos el _Pipeline_ para que use Jwt.

```csharp
//...

public static void Main(string[] args)
{
    // ...
    // Cargamos los settings en una variable para usar la configuración
    var jwtSettings = builder.Configuration.GetSection("JwtSettings");
    var secretKey = jwtsettings["SecretKey"]!;

    // Configuramos el builder
    builder.Services.AddAuthentication(options =>
    {
        options.DefaultAutheticateScheme = JwtBearerDefaultsAuthenticationScheme;
        options.DefaultChallengeScheme = JwtBearerDefaults.AutheticationScheme;
    })
    .AddJwtBearer(options => 
    {
        options.TokenValidationParameters = new TokenValidationParameters
        {
            ValidateIssuer = true,
            ValidateAudience = true,
            ValidateLifetime = true,
            ValidateIssuerSigningKey = true,
            ValidIssuer = jwtSettings["Issuer"],
            ValidAudience = jwtSettings["Audience"],
            IssuerSigningKey = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(secretKey)),
        };
    });
    // ...
}
```

Para manejar el login necesitamos una clase. En `Models/LoginRequest.cs`.

```csharp

// Es una clase POCO, dos propiedades públicas y automáticas

{

    public string Password { get; set; }
```

Volvemos a `AuthController.cs`

```csharp
[ApiController]
[Route("[controller]")]
public class AuthController : ControllerBase
{
    // Creamos una variable privada para la configuración
    private readonly IConfiguration _configuration;
    
    // Un constructor
    public AuthController(IConfiguration configuration)
    {
        _configuration = configuration;
    }
    
    // Este controlador debe recibir una solicitud de tipo POST
    // Especificamos la ruta entre paréntesis
    [HttpPost("login")]
    public ActionResult Login([FromBody] LoginRequest request)
    {
        // Si no cumple con estos requisitos, no está autorizado
        if (request.Email != "admin@example.com" || request.Password != "qwerty")
        {
            return Unauthorized();
        }
        
        // Declaramos los claims para el token
        var claims = new[]
        {
            new Claim(ClaimTypes.NameIdentifier, "user-id-123"),
            new Claim(ClaimTypes.Name, request.Email),
            new Claim(ClaimTypes.Role, "Admin"),
        };
        
        // Ahora que tenemos los claims, armamos el token
        var jwtSettings = _configuration.GetSection("JwtSettings");
        var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(jwtSettings["SecretKey"]!));
        var creds = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);
        
        var token = new JwtSecurityToken(
            issuer: jwtSettings["Issuer"],
            audience: jwtSettings["Audience"],
            claims: claims,
            expires: DateTime.UtcNow.AddMinutes(60),
            signingCredentials: creds
        );
        
        // Hacemos un string del token
        var tokenString = new JwtSecurityTokenHandler().WriteToken(token);
        
        // TODO: completar esto
        return Ok();
    }
}
```

Todo el paso a paso está en el apunte.

Los token jwt se pueden verificar en `jwt.io`. Son fácilmente traducibles. Se
validan con el secreto que incorporamos a `appsettings.json`. Los claims no
están  encriptados. Está encriptado el secreto. Como el sistema tiene el
secreto, puede  validar los tokens.

# REST

Protocolo HTTP. Protocolo que tiene sus años. Podemos aprovechar HTTP o
ignorarlo.

Cuando se empezaba a usar para transimitir información entre servicios surgieron
diferentes estándares. Uno es SOAP.

SOAP. Usa HTTP como un canal para enviar y recibir información codificada en el 
cuerpo de la solicitud. Solo utiliza HTTP como transporte. No le interesaba la 
especificidad del protocolo. Como los verbos o los encabezados.

Pero nosotros podemos aprovechar HTTP. Por ejemplo, las distintas rutas. Para 
responder diferente de acuerdo a la ruta. También podemos tener rutas que
reciban POST. Es decir, usamos un verbo del protocolo y aprovechamos la ruta.

Lo último es _Hipermedia_. Es el nivel más alto. Un documento HTML no tiene solo
información. Sino también enlaces `<a href>`. Un formulario `<form>`. Imágenes.
`<img>`. No necesitamos saber nada de rutas. Solo hacemos click y listo.
Hipermedia es llevar esos servicios a HTTP. 

Una aplicación REST no solo enviaría información sobre los recursos solicitados.
Sino también vínculos a otras cosas.

Todo esto viene de `Designing Evolvable Web APIs with ASP.NET`.

REST es un estilo. Pone restricciones. Cliente-servidor. Sin estado. Caché. Y 
otros aspectos. Mientras más implementemos esto, más fácil va a ser de mantener
nuestro sistema. Más fácil va a evolucionar sin romper los clientes.

Existen alternativas a REST.

# Alternativas a REST

GraphQL. Creada por Facebook. El cliente, además de enviar solicitudes, puede
especificar qué información quiere. Capaz no nos interesa toda la información
de un registro. Esto no se puede evitar en REST. Porque hay un único punto de 
acceso. GraphQL permite especificar información dentro del recurso.

gRPC. Desarrollado por Google. Se aleja del estándar HTTP. Tiene que ver con 
arquitectura de microservicios. Los datos son binarios. Gana en rendimiento.
Usa HTTP/2 para transmitir datos binarios.
